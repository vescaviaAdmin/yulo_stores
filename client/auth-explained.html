1. Problem kya tha?

Pehle project mein authentication ka koi proper structure nahi tha. Sab kuch App.jsx (3500+ line ki ek file) mein thusa hua tha. Kuch problems thi:

❌ Pehle kya tha
  • Token localStorage mein store hota tha — hacker steal kar sakta hai (XSS attack)
  • Login screen har portal mein alag-alag inline thi — duplicate code
  • API calls directly fetch() se — koi interceptor nahi, koi auto-refresh nahi
  • Legacy endpoints jaise /restaurant_owner/login — jo server pe exist hi nahi karte
  • Koi role-based route protection nahi — koi bhi kisi bhi page pe ja sakta tha
  • Chef aur Waiter dono ek hi login se — no separation
✅ Ab kya hai
  • Access token memory mein — safer
  • Alag-alag clean login pages per role
  • Axios with interceptors — auto token refresh
  • Real backend endpoints use ho rahe hain
  • Protected routes — unauthenticated user redirect hota hai
  • Waiter aur Chef ke liye alag staffToken flow

3. api/client.js — The HTTP Engine

📄 src/api/client.js

Axios ek popular HTTP library hai. Simple fetch() ki tarah kaam karta hai lekin zyada features ke saath. Hamne ek single Axios instance banaya hai poori app ke liye.

💡

Instance kyun? Agar aap seedha axios.get() use karo, toh har request mein baseURL, headers sab manually likhna padega. Instance banao toh ek baar set karo, sab jagah automatically lagta hai.

3.1 Token Storage — Sabse Important Security Decision

❌ Bad Practice — localStorage
Pehle ka code (insecure)
// ❌ GALAT — hacker steal kar sakta hai
localStorage.setItem("token", accessToken)

// Koi bhi JS code (even 3rd party)
// localStorage read kar sakta hai
// Isko XSS attack kehte hain
  • JavaScript se accessible — XSS attack mein steal hota hai
  • Browser close hone ke baad bhi rehta hai — unnecessary
✅ Good Practice — Memory
Hamara code (secure)
// ✅ SAHI — sirf RAM mein
let _accessToken = null;

export function setAccessToken(token) {
  _accessToken = token;  // memory mein
}

// Page refresh pe gone — intentional!
// Isliye refresh token cookie use hoti hai
  • JavaScript se directly accessible nahi (no XSS risk)
  • Page refresh pe automatically clear — safer lifecycle

3.2 Staff Token — SessionStorage

Staff (waiter/chef) ka token sessionStorage mein rakhte hain, localStorage mein nahi:

StorageKab Khatam Hota HaiUse Case
memory Page refresh pe Owner access token — short-lived (15 min)
sessionStorage Tab close pe Staff token — POS terminal tab open rehti hai
localStorage Manually clear hone pe User profile info (sensitive nahi)
🎯

Real restaurant mein socho: Ek POS tablet pe waiter ka tab din bhar khula rehta hai. Tab close nahi karte. Tab close hone pe sessionStorage saaf ho jaata hai — security good. Lekin page refresh pe token rehta hai — UX good.

3.3 Request Interceptor — Token Automatically Lagana

Interceptor ek middleware hai — har request ke "beech mein" kuch karta hai. Jaise airport security — har flight se pehle check hota hai.

Request Interceptor — request jane se pehle
client.interceptors.request.use((config) => {
  // config.staff = true ho toh staff token,
  // warna owner token lagao
  const token = config._staff
    ? getStaffToken()
    : getAccessToken();

  if (token) {
    config.headers.Authorization = `Bearer ${token}`;
  }

  return config;  // modified request server pe jaata hai
});

// Ab tumhe KABHI likhna nahi padega:
// headers: { Authorization: `Bearer ${token}` }
// Sab automatic!

3.4 Response Interceptor — Auto Token Refresh

Yeh sabse complex lekin most important part hai. Samjho — access token sirf 15 minute ke liye valid hai. Kya user 15 minute mein automatically logout ho jaaye? Nahi! Silently refresh hona chahiye.

Token Expired Flow
API Call karo
→
401 TOKEN_EXPIRED
→
Interceptor pakadta hai
↓
POST /auth/refresh cookie se
→
Naya token milta hai
→
Original request dobara bhejo
Response Interceptor — simplified explanation
client.interceptors.response.use(
  (res) => res,   // success: seedha return kar do

  async (err) => {
    const code = err.response?.data?.code;

    if (code === "TOKEN_EXPIRED") {
      // Server ne bola token purana hai
      const { data } = await axios.post("/api/auth/refresh");
      // HttpOnly cookie automatically server pe gayi
      // Server ne naya token diya

      setAccessToken(data.data.accessToken);

      // Original request dobara bhejo — user ko pata nahi chalega
      return client(originalRequest);
    }

    return Promise.reject(err);
  }
);
⚠️

Ek edge case: Agar multiple requests same time pe 401 paaye, toh sab simultaneously refresh try karenge — refresh storm. Hamare code mein _queue aur _refreshing flag se yeh handle kiya hai — pehla request refresh karta hai, baaki wait karte hain.

4. api/auth.api.js — API Functions

📄 src/api/auth.api.js

Yeh file sirf API calls define karti hai — koi logic nahi, koi state nahi. Isko API Module pattern kehte hain.

auth.api.js — poori file
import client from "./client";

export const authApi = {
  // Owner / Customer ke liye
  signup: (body) => client.post("/auth/signup", body),
  login:  (body) => client.post("/auth/login", body),
  logout: ()     => client.post("/auth/logout"),
  refresh: ()    => client.post("/auth/refresh"),

  // Staff ke liye (PIN-based)
  staffLogin:  (body) => client.post("/staff/auth/login", body),
  staffLogout: ()     => client.post("/staff/auth/logout", {},
    { _staff: true }),  // interceptor ko batao staff token use karo
};
💡

Separation of Concerns — yeh ek important design principle hai. Har file sirf ek kaam kare. auth.api.js sirf "kaunsa endpoint call karna hai" jaanta hai — "state kya update hogi" yeh nahi jaanta. State ka kaam context ka hai.

5. OwnerAuthContext.jsx — Owner State

📄 src/context/OwnerAuthContext.jsx

Context kya hota hai?

React Context ek global state mechanism hai. Problem yeh thi ki React mein by default state sirf parent → child jaati hai (props drilling). Context se aap kisi bhi component se state access kar sakte ho bina props pass kiye.

❌ Bina Context (Props Drilling)
// App → Layout → Sidebar → Button
// Har jagah user pass karo
<Layout user={user}>
  <Sidebar user={user}>
    <UserButton user={user} />
  </Sidebar>
</Layout>
✅ Context se (Direct Access)
// Kisi bhi component mein directly:
function UserButton() {
  const { user } = useOwnerAuth();
  // No props needed!
}

Page Refresh ka Problem — aur Solution

Jab user page refresh karta hai, React state reset ho jaati hai. Token (jo memory mein tha) gone. Lekin user ko fir se login karne ko force nahi karna chahiye — bad UX hai.

Page Refresh ke baad kya hota hai
User refreshes page
→
Memory token = null
→
Context mounts
↓
localStorage mein profile hai?
→
POST /auth/refresh (cookie)
→
Naya token mila!
OwnerAuthContext — auto-refresh on mount
useEffect(() => {
  // Sirf mount pe chalega ([] dependency)
  if (user && !getAccessToken()) {
    // User ka profile localStorage mein hai
    // lekin token memory mein nahi (page refresh)
    authApi.refresh()
      .then(({ data }) => setAccessToken(data.data.accessToken))
      .catch(() => {
        // Cookie bhi expire ho gayi — force logout
        setUser(null);
        localStorage.removeItem("yulo_owner_profile");
      })
      .finally(() => setLoading(false));
  } else {
    setLoading(false);
  }
}, []);   // empty array = sirf mount pe
💡

HttpOnly Cookie kya hai? Server ek special cookie set karta hai Set-Cookie: refreshToken=...; HttpOnly. HttpOnly ka matlab hai JavaScript is cookie ko read nahi kar sakta — sirf browser hi automatically server pe bhejta hai. Isliye yeh super secure hai!

6. StaffAuthContext.jsx — Staff State

📄 src/context/StaffAuthContext.jsx

Staff (waiter aur chef) ke liye alag context hai. Reason:

  • Alag token type: Staff ke liye staffToken hai, owner ke liye accessToken
  • No refresh: Staff token expire ho toh dobara PIN daalo — koi silent refresh nahi
  • Session-scoped: POS tablet pe tab band hone par automatically logout — good security for shared devices
StaffAuthContext — login function
const login = useCallback(async ({ restaurantId, pin }) => {
  // Server pe PIN verify hoga
  const { data } = await authApi.staffLogin({ restaurantId, pin });
  const { staff: s, staffToken } = data.data;

  // Token sessionStorage mein rakhna
  setStaffToken(staffToken);  // module-level function

  // Profile sessionStorage mein
  const profile = { ...s, restaurantId };
  sessionStorage.setItem("yulo_staff_profile",
    JSON.stringify(profile));

  setStaff(profile);
  return profile;
}, []);
🎯

useCallback kyun use kiya? Agar login function directly StaffAuthContext mein define karo without useCallback, toh har re-render pe naya function banata hai. Jo context consume karta hai woh unnecessary re-render hota hai. useCallback ensure karta hai ki function tab hi naya bane jab dependencies change ho.

7. Login Screens

7.1 OwnerLoginPage — /owner/login

📄 src/screens/auth/OwnerLoginPage.jsx

Clean white card design, dark background. Login + Signup tabs (mode switcher).

Key pattern — form submit handler
async function handleSubmit(e) {
  e.preventDefault();  // page reload mat karo
  setError("");         // purana error clear karo
  setLoading(true);

  try {
    if (mode === "login") {
      await login({ email: form.email, password: form.password });
    } else {
      await signup({ name: form.name, ...form });
    }
    // Success: dashboard pe bhejo
    navigate("/dashboard", { replace: true });
    // replace: true = back button login pe nahi jaaye
  } catch (err) {
    setError(err.message);  // error show karo
  } finally {
    setLoading(false);     // hamesha loading false karo
  }
}
⚠️

finally kyun zaroori hai? Agar try ya catch mein error aaye, finally phir bhi chalega. Agar setLoading(false) sirf try mein likho aur error aaye, button hamesha disabled rehega. User dobara submit nahi kar payega!

8. Protected Routes — Route Guards

📄 src/components/OwnerRoute.jsx   src/components/StaffRoute.jsx

Ye components ensure karte hain ki koi bhi unauthenticated user protected pages pe na jaaye. Isko Route Guard pattern kehte hain.

OwnerRoute — simplified
export default function OwnerRoute({ children }) {
  const { isAuthenticated, loading } = useOwnerAuth();

  // Page refresh pe token refresh ho raha hai — wait karo
  if (loading) return <LoadingScreen />;

  // Not logged in — login page pe bhejo
  if (!isAuthenticated)
    return <Navigate to="/owner/login" replace />;

  // Sab theek hai — actual page show karo
  return children;
}
StaffRoute — role bhi check karta hai
export default function StaffRoute({ children, role }) {
  const { isAuthenticated, staff } = useStaffAuth();

  if (!isAuthenticated)
    return <Navigate to="/staff/login" replace />;

  // Chef ne waiter URL open kiya — wrong portal redirect
  if (role && staff?.role !== role) {
    const target = staff?.role === "waiter"
      ? "/waiter" : "/chef";
    return <Navigate to={target} replace />;
  }

  return children;
}
🎯

replace kyun? <Navigate to="/login" replace /> mein replace se browser history mein protected URL store nahi hota. Bina replace ke, user back button press kare toh protected page pe wapas aa jayega (jo phir redirect karega — annoying loop).

9. main.jsx — Sab Wrap Karna

📄 src/main.jsx

Provider Pattern: React mein context aur libraries ko "wrap" karna padta hai root level pe. Jo bhi andar hai woh sab access kar sakta hai. Socho jaise building ka electrical board — wiring ek jagah, sab rooms mein current.

main.jsx — Provider tree
<QueryClientProvider client={queryClient}>   // TanStack Query
  <BrowserRouter>                         // React Router
    <OwnerAuthProvider>                     // Owner auth state
      <StaffAuthProvider>                   // Staff auth state
        <App />                             // Poori app
      </StaffAuthProvider>
    </OwnerAuthProvider>
  </BrowserRouter>
</QueryClientProvider>

// Order matters!
// BrowserRouter must be around everything using useNavigate/useLocation
// Auth providers must be inside BrowserRouter (they use navigate internally)
⚠️

Order ka dhyan rakho: BrowserRouter bahar hona chahiye auth providers se. Kyunki auth context mein useNavigate use ho sakta hai — aur useNavigate sirf BrowserRouter ke andar kaam karta hai.

11. Good vs Bad Practices

Security

✅ Good Practices (Jo Humne Kiya)
  • Access token in memory — XSS attack se safe
  • Refresh token HttpOnly cookie — JavaScript access nahi kar sakta
  • Staff token sessionStorage — Tab close = auto logout (shared POS devices ke liye safe)
  • Token blacklisting — Server Redis pe logout ke baad token invalidate karta hai
  • Short token lifetime — 15 min access, 7 din refresh
❌ Bad Practices (Jo Avoid Kiya)
  • Token localStorage mein — koi bhi JS code read kar sakta hai
  • Long-lived access tokens — agar steal ho toh zyada damage
  • Token URL mein — browser history mein store hota hai
  • Same token for all roles — compromise ek ka, sab expose

Code Architecture

✅ Good Practices
  • Separation of Concerns — API layer, state layer, UI layer alag
  • Single Axios instance — ek jagah se sab manage
  • Custom hooks (useOwnerAuth) — context access clean way mein
  • Error boundary — context bahar use hone pe clear error
  • Loading state — page refresh pe blank screen nahi
❌ Bad Practices
  • Monolithic component — 3500 line ka App.jsx (legacy)
  • Direct fetch() calls in components — no centralized error handling
  • Hardcoded tokens in components — maintenance nightmare
  • No loading state on refresh — flash of wrong content

UX Practices

✅ Good UX
  • Login ke baad replace: true — back button loop nahi
  • finally mein loading false — button hamesha clickable
  • Error message show karo — user ko pata chale kya galat hua
  • PIN field: numeric only enforce — keyboard automatically numeric aata hai mobile pe
  • Loading state on submit — double submit prevent
❌ Bad UX
  • Error nahi dikhana — user confused
  • Loading state nahi — user multiple baar click karta hai
  • Page reload on form submit — browser default behavior
  • Token expire hone pe seedha login — user ka kaam lost

12. Key Concepts jo Seekhe

JWT — JSON Web Token

Token ek encoded string hoti hai jisme user ki info hoti hai. Server ise verify karta hai apni secret key se. Koi DB lookup nahi chahiye — fast aur stateless. Hamare system mein do tokens hain: short-lived access token (15 min) aur long-lived refresh token (7 din).

Axios Interceptors

Request interceptor — request jane se pehle modify karo (token lagao). Response interceptor — response aane ke baad handle karo (401 pe refresh karo). Middleware pattern ka React/browser equivalent.

React Context + Custom Hook Pattern

Context banao (createContext) → Provider component mein wrap karo → Custom hook (useOwnerAuth) se consume karo. Direct useContext(OwnerAuthContext) use karne ke bajay custom hook better hai — error message clear milta hai, aur future mein implementation change karna easy.

XSS (Cross-Site Scripting)

Ek attack jisme hacker website pe malicious JavaScript inject karta hai, phir localStorage.getItem("token") se token steal karta hai. Isliye access token kabhi localStorage mein nahi rakhte. Memory (variable) mein rakho — JS inject ho bhi jaaye toh closure scope se outside nahi nikal sakta (theoretically safer).

HttpOnly Cookie

Server Set-Cookie: refreshToken=...; HttpOnly; SameSite=Strict response mein bhejta hai. HttpOnly = JavaScript window se nahi padha ja sakta. SameSite=Strict = sirf same site ki requests pe cookie jayegi — CSRF attack se bachav. Browser automatically ise har request pe server pe bhejta hai.

Role-Based Access Control (RBAC)

Har user ka ek role hota hai: customer, restaurant_owner, waiter, chef. Protected routes role check karte hain. Ek waiter owner dashboard nahi dekh sakta, ek chef waiter portal nahi use kar sakta — even if URL directly dalo.

Optimistic vs Loading State

Loading state = action hone se pehle UI disable karo (button disabled, spinner show karo). Prevents double-submit. Optimistic update = assume success, show result immediately, revert on failure — faster feel (TanStack Query mein use hoga later). Dono important hain different scenarios mein.
